Motion Planning

PRM with parameters configurable via variables inside PRM.cpp. Liam's camera was used. Some difficulty I had when trying to program this was primarily getting the agents moving with boids. The parameters were hard to tune without it bouncing back and forth. Other difficulty was finding that parentheses I missed when checking for intersections. Another difficulty is OpenGL in general. I could have used processing, but I wanted to figure out how OpenGL works because it's powerful. From my previous project, I came in with misunderstandings, so the draw function in PRM.cpp keeps bufferring new vertices instead of reusing them.

All agents use the same path or the reverse path. More can be generated by creating another configuration object.
Path smoothing is in all of it.
Boids is used and can be seen in all of the scenarios.
Extent of agents is 0.5m and is used, but not as a cylinder or circle. If it doesn't seem like it, divide radius on line 150 of agent.cpp by 2.f
Problems with highly dense scenarios and tight paths, such as the two circles on the bottom.
The following were implemented:

  • Boids
  • PRM with UCS
  • Took into account agent extents, which are 0.5m and can be configured. Seen by the gap between points
  • Multiple Obstacles
  • Scenarios with interesting agent interactions, along the same path, colliding, and gathering once stopped
  • Scenarios with interesting local minima
  • Scenario where simulation produces odd behavior: where highly dense and forces push agents away from path and they get stuck because they don't replan
  • Path smoothing
  • I think my program can do 3D navigation since it only uses spheres, but I never tested it out, so whatever
